# Design System — Gmail Filter Cleanup

Generated by /plan-design-review on 2026-03-19
All decisions are intentional. When in doubt, ask before deviating.

---

## Design Philosophy

This is a **utility tool with a strong aesthetic point of view** — not a neutral dashboard. Users trust it
with their email settings, so every design decision either builds or erodes that trust. The visual style
should feel crafted, opinionated, and specific — not generated.

The goal of the "done" state is for the user to feel **empowered and in control** — they reviewed the
changes, they understood them, they approved them. Not "the AI cleaned up your mess" but "you simplified
your filters, with AI assistance."

---

## Color System — Dark by Default

The app is **always dark**. No light mode, no system preference detection, no toggle. This is a deliberate
aesthetic choice, not a missing feature.

### Base Palette

| Role | Tailwind class | Hex |
|------|---------------|-----|
| Page background | `bg-gray-950` | `#030712` |
| Nav / section bg | `bg-gray-900` | `#111827` |
| Card background | `bg-gray-800` | `#1f2937` |
| Elevated card | `bg-gray-750` (custom) / `bg-gray-800` | — |
| Border / divider | `border-gray-700` | `#374151` |
| Border subtle | `border-gray-800` | `#1f2937` |

### Typography on Dark

| Role | Tailwind class | Use |
|------|---------------|-----|
| Primary text | `text-gray-100` | Headings, body |
| Secondary text | `text-gray-400` | Subtitles, metadata |
| Tertiary / disabled | `text-gray-600` | Placeholders, footnotes |
| Link | `text-blue-400 hover:text-blue-300` | Inline links |

**Contrast requirement:** All body text must pass WCAG AA (4.5:1 ratio). Avoid `text-gray-500` on
`bg-gray-900` — it fails AA. Use `text-gray-400` minimum on dark backgrounds.

### Semantic Colors (dark variants)

| Signal | Background | Text | Border |
|--------|-----------|------|--------|
| Error | `bg-red-950/60` | `text-red-400` | `border-red-800` |
| Warning | `bg-yellow-950/60` | `text-yellow-400` | `border-yellow-800` |

| Neutral | `bg-gray-800` | `text-gray-300` | `border-gray-700` |

### Primary Action

- Primary button: `bg-blue-600 hover:bg-blue-500 text-white` (blue-600 has sufficient contrast on dark)
- Disabled: `bg-gray-700 text-gray-500 cursor-not-allowed`
- Destructive: `bg-red-700 hover:bg-red-600 text-white`

## Component Specs

### Badges

Badges carry both **color AND an icon prefix** to be accessible to colorblind users.

| Badge | Icon | Dark classes |
|-------|------|-------------|
| Dead label | `⚠` | `bg-red-950/60 text-red-400 border border-red-800` |
| Duplicate | `=` | `bg-yellow-950/60 text-yellow-400 border border-yellow-800` |
| High confidence | — | `bg-green-950/60 text-green-400` |
| Medium confidence | — | `bg-yellow-950/60 text-yellow-400` |
| Low confidence | — | `bg-orange-950/60 text-orange-400` |

### Step Navigation

The audit flow shows **4 named steps** as dots in the top nav:

```
Gmail Filter Cleanup                    Sign out
───────────────────────────────────────────────
  ● Audit  ○ Enter Key  ○ Review  ○ Apply
```

- Filled dot (`●`) = current or completed step
- Empty dot (`○`) = upcoming step
- The two loading states (consolidating, applying) are **not** separate nav steps —
  "consolidating" shows as a transition within "Enter Key", "applying" as a transition within "Apply"
- Step labels: `text-xs text-gray-400`, active step label: `text-xs text-gray-100 font-medium`
- Dots: 6px circles, `bg-gray-100` (complete), `bg-gray-600` (upcoming)

### Google Sign-In Button

Keep the Google sign-in button in its **light/white variant** even in dark mode. This preserves
Google's brand recognition and avoids unofficial dark-mode button styling. White button on dark
background creates a deliberate visual contrast that draws attention to the CTA.

### Loading States

All loading states show a spinner (`border-blue-500 border-t-transparent`) + a message.

- Short loads (< 5s): spinner + message only
- Long loads (15-30s): spinner + message + italic note below ("This may take 15–30 seconds…")
- The `consolidating` step note must set expectations: users should not be alarmed by the wait

## Information Architecture

### Landing Page (`/`)

Hierarchy:
1. Title: "Gmail Filter Cleanup" — `text-2xl font-semibold text-gray-100`
2. Subtitle: value prop copy — `text-gray-400`
3. **Before/After example**: `47 filters → 12 filters` — real anonymized data from a completed run,
   labeled `(example from a real cleanup)` in `text-xs text-gray-600 italic`
4. Sign-in button (Google, light variant)
5. Trust note: "Reads and writes Gmail filter settings only. No email content is accessed."
6. **Anthropic API key note**: "AI consolidation uses Claude — you'll need an Anthropic API key.
   Get one free at [console.anthropic.com](https://console.anthropic.com)." — `text-xs text-gray-500`
7. "What it does" card

The Anthropic API key note (item 6) **must appear on the landing page**, before the user signs in,
so the mid-flow friction of the key entry step is expected rather than surprising.

### Audit Page (`/audit`) — Step Flow

The 4-step nav is persistent across all steps. Each step has a clear heading and back navigation.

Steps visible in nav:
1. **Audit** — see filter inventory with dead/duplicate flags
2. **Enter Key** — provide Anthropic API key (with link to get one)
3. **Review** — approve/reject each proposal (diff view)
4. **Apply** — apply + undo

## Copy & Tone

**Voice:** Direct, honest, a little dry. Not cheerful. Not "AI magic!" — more "here's what this does."

### Key copy decisions

| Moment | Copy |
|--------|------|
| Empty state (0 filters) | "You have no Gmail filters yet — looks like you're all set." |
| API key error (auth failure) | "[error message]\n\nGet your API key at [console.anthropic.com/keys](https://console.anthropic.com/keys)" |
| Session expired banner | "Your session expired. Please sign in again." (amber, already implemented) |
| Done state headline | "Your filters have been reviewed and simplified." |
| Done state subhead | "You consolidated N rules into M — you know exactly what's in your inbox now." |
| Done state counts | "{M} filters created · {N} deleted" (centered, `text-gray-400 text-sm`) |

### Done State — Full Design

The done state should communicate **empowerment and ownership**, not just task completion.
The user reviewed this, approved it, and understands their filter situation.

```
     ✔  Your filters have been reviewed and simplified.

  You consolidated 12 rules into 4 —
  you know exactly what's in your inbox now.

        4 filters created  ·  12 deleted

              [ ↩ Undo all changes ]
```

- Headline: `text-lg font-semibold text-gray-100`
- Subhead: `text-sm text-gray-400 max-w-sm mx-auto text-center`
- Counts: `text-sm text-gray-500`
- Undo button: prominent secondary style — NOT a small ghost button. Users should feel safe.

## Accessibility

- All interactive elements must have `aria-label` when the label isn't visually obvious
- The API key input: `aria-label="Anthropic API key"` (placeholder alone is not sufficient)
- Touch targets: minimum 44×44px for all buttons on mobile
- Badge color + icon: both must communicate the meaning (do not rely on color alone)
- Contrast: all body text ≥ 4.5:1 against its background (WCAG AA)
- Keyboard nav: standard browser defaults are acceptable; no custom focus traps needed

---

## Not In Scope

| Decision | Why deferred |
|----------|-------------|
| Dark/light toggle | Side project; dark-by-default is the aesthetic choice, not a missing feature |
| System `prefers-color-scheme` | Adds ~3x implementation complexity; deferred until user demand |
| Animation / transitions between steps | Step transitions can be instant; no need for page-slide animations |
| Illustration / iconography beyond badge icons | Not worth the asset overhead for a utility tool |
| Internationalization / RTL | Side project; English only |

## What Already Exists (do not reinvent)

- **Inter font** — already loaded via `next/font/google`. Keep.
- **`rounded-lg` radius** — consistent across all components. Keep.
- **`font-mono` for filter criteria** — excellent choice, already in place. Keep.
- **Diff view pattern** (before: red bg, after: green bg) — already in `ProposalCard`. Migrate to dark variants.
- **Badge component** — already exists in `audit-client.tsx`. Add icon prefix and migrate to dark colors.
- **Loading spinner** — already implemented. Migrate spinner color to `border-blue-500`.
